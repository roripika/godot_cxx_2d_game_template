# Stress / Scale 検証方針メモ

## 1. 現状と課題
Phase 2 までのフィットネステスト（Mystery, Billiards, Roguelike, Rhythm）は、いずれも「少数のエンティティ」と「低頻度〜中頻度の更新」を対象としてきました。
カーネルが数千のエンティティや秒間数万の WorldState 書き込みに耐えうるかを検証するフェーズが必要です。

## 2. 検証の柱

### 2.1 Sandbox Test モジュールの具体化
`sandbox_test` を「大量の状態管理」をテストする専用モジュールとして定義します。
- **検証項目**:
    - 数百個のタイル個別状態の保持と一括更新。
    - 大規模なシナリオグラフ（1,000ノード超）のメモリフットプリント。
    - `WorldState` の検索・書き込みコストの O(n) 特性。

### 2.2 Bullet Hell (弾幕) フィクスチャ
移動のみを行う大量の `BaseEntity` を管理する最小限のテスト。
- 衝突判定を伴わない「純粋な移動更新」のコストを測定。
- 衝突判定は Godot のエンジン側に任せ、Kernel はその「結果」をどう受け取るか（再入のオーバーヘッド）を確認。

## 3. フェーズ延期判断
以下の項目は、基盤フェーズの完了条件とはせず、Phase 3 以降の「スケーラビリティ改善」トラックへと延期します。
- **マルチスレッド `ScenarioRunner`**: 現在のシングルスレッド実行でも、論理更新のみであれば十分なパフォーマンスが出ると予測されるため。
- **WorldState の差分同期 (Network)**: ネットワーク越しの大規模同期はゲームエンジンレベルの難問であり、本プロジェクトの「AIがロジックを書く」という主目的からは一段階遠いため。

## 4. 軽量フィクスチャの導入案
Phase 2 の締めくくりとして、以下の軽量な負荷テストを `mystery_test` の拡張版として置くことを検討します。
- `mystery_stress_test.yaml`: 50以上のダイアログと分岐を高速で自動遷移させるだけのシナリオ。
- これにより、スタックオーバーフローやメモリリークの基本的なチェックを行います。
