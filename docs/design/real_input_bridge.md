# Real Input Bridge Design Memo

## 1. 目的
Karakuri Kernel v2.0 は「決定論的なシナリオ実行」を前提としています。一方、ユーザーからの実入力（キーボード、マウス、コントローラー）は非同期かつ非決定論的です。本メモでは、この両者を安全に接続するためのブリッジ構造を設計します。

## 2. 基本原則
1. **コア非変更**: `src/core/` のロジックを変更せず、Task 層で入力を解決する。
2. **決定論の維持**: WorldState に記録されるのは「未加工のシグナル」ではなく、解決済みの「論理コマンド」とする。
3. **副作用の局所化**: 入力待機によるブロッキングは、特定の `WaitInputTask` 内に閉じる。

## 3. アーキテクチャ構成

### 3.1 InputService の役割
既存の `InputService` を拡張し、以下の責務を持たせます。
- **Raw Input キャプチャ**: Godot の `_input(event)` をトリガーに、タイムスタンプ付きで入力を一時バッファに格納。
- **Intent 変換**: 特定のゲームモジュールが必要とする「論理入力（例: MoveUp, Attack）」に変換。

### 3.2 Input Buffer (Queue)
`InputService` 内に保持される、スレッドセーフな（またはメインスレッドで完結する）キュー。
- シナリオのステップ更新の合間に発生した入力を消失させない。
- `ScenarioRunner` の更新サイクル（`process`）の冒頭で、バッファから最新の確定入力を取り出す。

### 3.3 WaitInputTask (ゲートウェイ)
実入力を待つための標準タスク。
```yaml
- action: wait_input
  payload:
    valid_commands: ["move_up", "move_down", "attack"]
    timeout: 5.0
    store_key: "player:last_command"
```
- キューに有効なコマンドがあるまで `Running` を返す。
- コマンドを取得したら `WorldState` に書き出し、`Success` で次へ進む。

## 4. WorldState への記録ルール
- タイムスタンプや座標といった不連続な値は、そのまま WorldState に書き込まず、グリッド座標や離散化した方向へと正規化してから保存する。
- これにより、入力を再現（Replay）する際に、ハードウェアの微細な差異を吸収できる。

## 5. 次フェーズへの課題
- **多重入力の優先順位**: 同一フレームに複数の入力があった場合の解決ルール。
- **ロールバック耐性**: 将来的にステート巻き戻し機能を導入する場合、入力ログの再実行が必要。
- **UIとの競合**: UI操作（ボタンクリック）とゲーム内入力の排他制御。
