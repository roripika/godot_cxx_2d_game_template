# Karakuri アーキテクチャ保護：PRレビュー・ガイドライン

## 1. はじめに：Karakuri の鉄則
`Karakuri (src/core/)` は、いかなるゲームジャンルにも転用可能な「共通インフラ」です。
特定のゲーム（現在はミステリー）の文脈がコア層に混入することは、将来の拡張性を奪う「技術的負債」となります。

本ガイドラインでは、Pull Request (PR) 時や実装前に必ずチェックすべき **「Karakuri コア汚染を検知する 5つの警告サイン」** を定義します。

---

## 2. コア汚染の 5つの警告サイン

### ☢️ サイン 1：語彙（ドメイン）の汚染
**変数名やクラス名に、特定のゲームジャンルを想起させる単語が混ざっていないか？**

*   **❌ 危険なコード (In Core)**:
    ```cpp
    // 証拠品マネージャーという名前はミステリー専用
    class EvidenceManager : public Node { ... }; 
    void check_contradiction(); // 矛盾チェックという概念はコアには不要
    ```
*   **⭕ 健全なコード (In Core)**:
    ```cpp
    // 基盤では「IDを管理する抽象的なコンテナ」に徹する
    class ItemCatalog : public Node { ... };
    bool evaluate_condition(Dictionary params);
    ```
    > [!TIP]
    > そのコードを「RPG」や「アクションゲーム」に持っていったとき、クラス名を見て違和感があるなら、それは Mystery 層 (`src/mystery/`) に移すべきです。

### ☢️ サイン 2：UI・演出の直書き
**C++ 基盤側で直接 UI のプロパティ（色、ラベル、アニメーション）を操作していないか？**

*   **❌ 危険なコード (In Core)**:
    ```cpp
    // 文字が赤いのは「犯人のセリフ」だから。それはゲーム固有の演出。
    dialogue_label->set_modulate(Color(1, 0, 0));
    ```
*   **⭕ 健全なコード (In Core)**:
    ```cpp
    // コアは「イベント」を発火するだけ。見た目は UI (GDScript) が決める。
    emit_signal("dialogue_styled", "danger_red");
    ```

### ☢️ サイン 3：ハードコードされた「面白さ」
**「ゲームのルール（勝敗、バランス）」が C++ コードに直接書き込まれていないか？**

*   **❌ 危険なコード (In Core)**:
    ```cpp
    // 「3回ミスで死亡」は一つのルールに過ぎない
    if (miss_count >= 3) { trigger_game_over(); }
    ```
*   **⭕ 健全なコード (In Core)**:
    ```cpp
    // 基盤は「判定エンジン」を提供し、閾値はデータ (JSON) から読み込む
    if (ConditionEvaluator::evaluate(limit_condition)) { ... }
    ```

### ☢️ サイン 4：他ジャンル転用テストの失敗
**その機能を追加することで、他のジャンルを作る際に「邪魔な重り」にならないか？**

*   **❌ 危険なコード (In Core)**:
    ```cpp
    // 全ての Entity に「尋問（Interrogate）」メソッドを持たせる
    class BaseEntity {
        virtual void start_interrogation(); 
    };
    ```
*   **⭕ 健全なコード (In Core)**:
    ```cpp
    // 基本的な「インタラクト」のみを定義し、詳細は派生クラスで実装する
    class BaseEntity {
        virtual void on_interact(); 
    };
    ```

### ☢️ サイン 5：特定のアセットパスへの依存
**`res://` パスなどのリソース場所を C++ に直書きしていないか？**

*   **❌ 危険なコード (In Core)**:
    ```cpp
    // 特定の画像ファイルに依存している
    Ref<Texture2D> icon = loader->load("res://assets/mystery/ui/frame.png");
    ```
*   **⭕ 健全なコード (In Core)**:
    ```cpp
    // パスは Export 変数や設定ファイル（JSON/YAML）から注入する
    @export_file("*.png") var ui_frame_path: String
    ```

## 3. レビュー時のチェックリスト

PR を承認する前に、以下の 5 項目を自問自答してください。

1.  [ ] **コア層に `mystery` の文字が 1 つも含まれていないか？** （コメント内も含む）
2.  [ ] **このコードを一切変更せずに、明日「シューティングゲーム」を作れるか？**
3.  [ ] **視覚表現の操作を、シグナルやテーマ設定に逃がしているか？**
4.  [ ] **定数（ダメージ量、確率、パス）を、JSON などのデータ層から読み込んでいるか？**
5.  [ ] **クラスの責務は「仕組みの提供」に限定されており、「意味の定義」をしていないか？**

## 4. 最後に
アーキテクチャの境界線を守ることは、一時的に手間が増えるように見えます。しかし、これにより私たちは **「一つのソースコードから、無限のジャンルのゲームを生み出す機巧（Karakuri）」** を手に入れることができます。

コア汚染を発見したら、恐れずにリファクタリングを提案しましょう。
