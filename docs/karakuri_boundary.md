# Karakuri アーキテクチャ境界定義 (Architecture Boundary)

## 1. 概要
`Karakuri` は、あらゆる2Dゲーム（ミステリー、プラットフォーマー、ローグライク等）で使い回せる「共通インフラ層」です。
本作における最大のルールは **「Karakuri は実行器に徹し、ゲームの面白さ（意味・見た目）は各デモ層に委譲する」** ことです。

## 2. 迷った時の判定フローチャート

新しい機能を実装する際、以下の質問に**すべて「YES」**と答えられるものだけが `src/core/` (Karakuri) に入る資格を持ちます。

1. [ ] **普遍性**: Mystery（現在のデモ）という概念を完全に消し去っても、その機能は存在価値があるか？
2. [ ] **汎用性**: 他のジャンルのゲームでも、全く同じクラス名・メソッド名で使えるか？
3. [ ] **構造重視**: 「見た目（UI/演出）」ではなく、「仕組み（システム）」であるか？
4. [ ] **独立性**: 特定のゲーム専用の語彙（例: `evidence`, `testimony`）を含んでいないか？
5. [ ] **テスト容易性**: その機能単体で、UIを介さずにC++レベルの単体テスト（Contract Test）が書けるか？

---

## 3. 実装テクノロジーの選定ルール (C++ vs GDScript vs Node)

Karakuri基盤はパフォーマンスと堅牢性を重視し **C++ (GDExtension)** で構築しますが、すべてをC++で書くのはGodotの強みを殺します。以下の基準で技術を選定します。

* **C++ (GDExtension)** にすべきもの:
  * 状態管理、セーブ/ロード、シナリオのパース、タスクランナー、重い計算処理。
* **Godot Resource (.tres / C++ Resource継承)** にすべきもの:
  * 永続設定、シナリオ定義データ、アイテムのマスタデータ（YAMLから変換された中間データ）。
* **GDScript / Scene Node (.tscn)** にすべきもの:
  * UIのレイアウト、アニメーション（AnimationPlayer）、画面固有の入力ハンドリング、パーティクル制御。

## 4. コンポーネント別 判定表

### 🟢 Karakuri (src/core/) に【入れる】もの
ゲームの土台となる「作品の違いが出ない部分」。

| コンポーネント | 説明 | 実装方針 |
| :--- | :--- | :--- |
| **Save / Load** | 保存先、スロット、暗号化などの共通化 | C++ (静的メソッド または Autoload) |
| **Localization** | 多言語切替、CSV/JSONからの辞書ロード | C++ (Autoload) |
| **SceneFlow** | `push`, `pop`, `replace` 等の画面遷移API | C++ (Autoload) |
| **ScenarioRunner** | YAML/JSONを読み込み、順次アクションを発火する実行器 | C++ (Node) |
| **TaskRunner** | 非同期処理、スキップ、ロールバックの基盤 | C++ (RefCounted / Node) |
| **Debug Hooks** | FPS表示、フラグ確認、E2Eテスト用の入力注入基盤 | C++ |

### 🔴 各ゲーム層 (src/mystery/ 等) に【逃がす】もの
作品性が強く、ジャンルや手触りに直結する部分。

| コンポーネント | 説明 | 実装方針 |
| :--- | :--- | :--- |
| **Mystery固有Action** | 証拠品の入手、尋問UIの起動、カメラの特定ズーム | C++ (KarakuriのActionを継承) |
| **ドメインモデル** | `Evidence` (証拠), `Testimony` (証言) などの状態管理 | C++ または GDScript |
| **UI・演出** | 会話ウィンドウの見た目、証拠品リストUI、HPバー | Scene (.tscn) + GDScript |
| **プレイヤー操作** | 調査モードのクリック判定、キャラの移動ロジック | Scene内のNode |
| **ジャンル専用ルール** | 推理の正誤判定、コンボ計算など | C++ または GDScript |
