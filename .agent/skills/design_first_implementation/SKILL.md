# 🛠️ AgentSkills: Design-First Implementation

今後、新しい要件（Requirements）が提示された際、いきなり実装コード（.cppや.gdなど）を書いてはならない。必ず以下の手順を踏むこと。

## 1. 設計ドキュメント（Design Document）の作成
コードを書く前に、`docs/design/` 配下に以下の構成で Markdown ファイルを作成し、ユーザー（最終承認者）のレビューを受けること。
M4のパワーを活かし、**Mermaid記法**を用いた図解を積極的に取り入れること。

* **Objective (目的):** その機能がなぜ必要か、何を解決するか。
* **Data Architecture (データ構造):** 使用する Enum, Struct, クラスの定義。
* **Logic Flow (論理フロー):** Mermaid記法を用いた状態遷移図（State Machine）やフローチャート。
* **Impact Assessment (影響範囲):** 既存のどの機能と競合するか、デグレのリスクはどこか。

## 2. ヘッダーコメントへの「意図（Why）」の埋め込み
生成するC++ファイルの冒頭に、必ず設計ドキュメントへのリンクと、そのコードが「なぜこの設計になったか」という意図を要約して記述すること。

## 3. テスト仕様（Test Cases）の定義
「どうなれば成功か」というテスト条件をドキュメントに明文化し、それに合格するスモークテスト/E2Eテストを同時に設計すること。

**※ 承認が出るまで実装（.cpp等の編集）を開始してはならない。コードを修正した際は、必ず設計ドキュメントも同期して更新すること。**
